## 1. 版本变更说明

### 1.1 v0.1 的核心定义

v0.1 将产品定义为：

> 基于 Skill 的可解释投研决策系统。

其核心范式是：

```text
模型给证据；
Skill 定方法；
Agent 做推理；
流程管决策；
链路保透明。
```

### 1.2 v0.2 的核心修正

v0.2 对 v0.1 进行重要修正：

> Skill 不应被定义为产品中心。Skill 是 Agent 能力封装、领域方法沉淀和工具使用说明的一种形式，但不是整个系统的唯一核心。

v0.2 将产品中心调整为：

> 可编排、可审计、可评估的 Agentic 投研决策工作流。

新的核心范式是：

```text
数据提供上下文；
模型和工具生成证据；
编排器选择执行路径；
Workflow 和 Agent 完成分析；
投资手册塑造判断；
护栏和评估控制风险；
Decision Trace 建立信任。
```

### 1.3 关键变化

v0.2 相比 v0.1 的核心变化包括：

1. 从 **Skill-centric** 转向 **Workflow / Orchestration-centric**；
    
2. 从“五个 Agent”调整为“五个分析能力域”；
    
3. 从“Agent 执行 Skill”调整为“编排器调度模型、工具、Workflow、Agent、审查器和人工节点”；
    
4. 从“用户自定义 Skill”调整为“用户选择或配置 Investment Playbook”；
    
5. 将 Skill 降级为 Capability Package 的组成部分；
    
6. 新增 Governance、Guardrails、Evaluation、Observability 等生产级 Agent 系统要素；
    
7. MVP 阶段暂不强调用户自由编辑 Skill，而优先验证透明投研工作流是否有价值。
    

### 1.4 v0.3 的核心修正

v0.3 的核心目标不是继续扩展产品细节，而是：

1. 收缩 SPEC-001 自身边界；
2. 澄清编排器与能力域的接口关系；
3. 明确关键概念的开放问题；
4. 将架构、能力域、Playbook、Evaluator、Decision Trace 等详细设计标记为迁移至后续 SPEC。

v0.3 相比于 v0.2 的核心变化包括：

1. 明确 **编排器调度能力域级别的 Analysis Domain Job**，不侵入能力域内部；
2. 新增 **Analysis Card** 作为能力域标准输出占位；
3. 声明 **Playbook 执行机制**为开放问题（SPEC-006）；
4. 区分 **Decision Trace（用户视角）与 Observability（运维视角）** 的受众和展示目标；
5. 明确 **MVP 优先面向宏观/中观敏感型投资者**，保留 Macro/Meso 但控制范围；
6. 约束 **MVP Evaluator 只做检查和标注，不触发自动重写**；
7. 承认 **Playbook 适用性局限**：系统早期无法判断 Playbook 本身是否适合当前市场环境；
8. 新增 **用户私有数据声明**和隐私边界；
9. 补充 SPEC-011（Case Library）和 SPEC-012（数据治理）至后续 SPEC 列表。

## 2. 背景

传统投研工作流中，投资者需要处理宏观环境、行业变化、公司基本面、公司事件、市场情绪、价格行为和风险约束等多类信息。

现有 AI 投研产品大多落在以下几类形态：

1. 新闻摘要工具；
    
2. 财报问答工具；
    
3. 股票分析 Chatbot；
    
4. 黑箱式买卖信号；
    
5. 多 Agent 投研团队模拟框架；
    
6. 量化指标看板；
    
7. 研报生成工具。
    

这些产品各有价值，但普遍存在以下问题：

- 信息处理和投资判断之间缺少清晰流程；
    
- Agent 或 LLM 容易自由发挥；
    
- 模型输出、事实证据和主观判断混在一起；
    
- 分析过程不透明，难以审计；
    
- 投资建议难以复盘；
    
- 用户自己的投资风格和决策流程难以产品化；
    
- 多个分析模块之间缺少统一编排；
    
- 系统经常要么过度自动化，要么只是简单问答；
    
- 缺少生产级 Agent 系统所需的护栏、评估和可观测性。
    

本产品希望从“AI 帮我分析股票”升级为：

> AI 按照可编排、可审计的投研工作流，辅助用户形成结构化投资判断。

## 3. 产品一句话定义

本产品是一个 **可编排、可审计的 Agentic 投研决策工作流系统**。

它通过数据连接器、模型、工具、工作流、Agent、投资手册、案例记忆、护栏、评估体系和 Decision Trace，帮助用户对股票或资产形成结构化、可解释、可复盘的投资判断。

本产品不是让 Agent 直接替用户做投资决策，而是让用户清楚看到：

```text
系统用了什么数据；
调用了什么模型和工具；
执行了什么工作流；
哪些 Agent 或分析模块参与了判断；
哪些证据支持；
哪些证据反对；
哪些规则或护栏被触发；
结论适合哪种投资风格；
哪些条件会让结论失效；
用户下一步应该检查什么。
```

## 4. 产品定位

### 4.1 产品是什么

本产品是一个面向投资研究和投资决策辅助的 Agentic Workflow System。

它的核心能力是：

1. 组织复杂投研任务；
    
2. 调用合适的数据、模型和工具；
    
3. 根据任务类型选择执行路径；
    
4. 使用 Workflow 和 Agent 生成结构化分析；
    
5. 按照投资风格或 Investment Playbook 形成决策建议；
    
6. 展示完整 Decision Trace；
    
7. 支持用户审阅、反馈和复盘。
    

### 4.2 产品不是什么

本产品不是：

1. 自动交易机器人；
    
2. 黑箱荐股系统；
    
3. 收益承诺工具；
    
4. 纯新闻摘要产品；
    
5. 纯财报问答产品；
    
6. 纯多 Agent 角色扮演系统；
    
7. 纯量化因子平台；
    
8. 直接替代持牌投顾或合规投资建议的系统；
    
9. 直接替用户决定买入、卖出或持仓的系统。
    

本产品的核心定位不是：

> 直接告诉用户买什么。

而是：

> 帮助用户按照清晰、透明、可复盘的投研流程，判断一项投资决策是否成立。

## 5. 核心用户

### 5.1 专业个人投资者

这类用户具备自主投资决策需求，愿意研究股票、行业和市场，但缺少系统化工具来组织自己的分析流程。

典型需求：

- 快速形成一只股票的结构化分析；
    
- 检查自己的投资逻辑是否完整；
    
- 获取反方观点；
    
- 按自己的投资风格判断是否值得买入、持有、等待或回避；
    
- 记录和复盘每次投资判断；
    
- 避免被单条新闻、社媒情绪或短期价格波动带偏。
    

### 5.2 小型投研团队

这类用户可能来自小型基金、家办、独立研究团队或投资社群。他们需要更高效地覆盖股票池，并生成标准化研究底稿。

典型需求：

- 标准化单股票研究流程；
    
- 对候选股票进行初筛和深度分析；
    
- 生成投委会前置材料；
    
- 保留分析链路和证据；
    
- 复盘历史判断质量；
    
- 在不同成员之间统一分析框架。
    

### 5.3 AI-native 投资工作流探索者

这类用户关注如何把 LLM、Agent、模型、工具和投资流程结合起来，可能希望基于本产品搭建自己的投研系统。

典型需求：

- 配置 Investment Playbook；
    
- 接入不同模型和数据源；
    
- 观察 Agentic Workflow 如何执行；
    
- 建立个人或团队的投资方法论资产；
    
- 迭代能力包、案例库和评估集。

## 6. 核心问题

本产品要解决的核心问题是：

> 投资者在面对复杂市场信息时，缺少一个可编排、可解释、可定制、可复盘的 AI 投研决策工作流。

具体拆解为以下问题。

### 6.1 信息过载

用户需要处理大量异构信息，包括：

- 宏观数据；
    
- 行业政策；
    
- 公司财报；
    
- 公司公告；
    
- 公司事件新闻；
    
- 社媒情绪；
    
- 新闻标题情绪；
    
- 技术指标；
    
- 价格行为；
    
- 资金流；
    
- 风险暴露；
    
- 用户持仓； <small>（用户私有数据）</small>
    
- 历史交易记录。 <small>（用户私有数据）</small>
    

单靠人工阅读效率低，单靠 Chatbot 又容易遗漏关键维度。

### 6.2 分析流程不稳定

普通 LLM 可以写出看似合理的股票分析，但不同时间、不同提示词下可能输出不同重点、不同判断标准和不同风险权重。

本产品需要通过 Workflow、Capability Package、Investment Playbook 和 Guardrails，让系统分析流程更加稳定。

### 6.3 执行路径缺少编排

不同投研任务不应该使用同一种处理方式。

例如：

```text
普通单股快评：适合固定 workflow；
财报后深度分析：适合财务模型 + 公告解析 + Agent 解释 + 审查器；
重大突发事件：适合事件识别 + 情绪分析 + 人工确认；
多空分歧明显：适合 Bull/Bear Debate + Risk Review；
用户自定义投资流程：适合 Playbook-driven Workflow。
```

本产品需要一个 Orchestration Layer，根据任务性质选择合适执行路径。

### 6.4 决策链不透明

投资建议如果只输出“买入 / 持有 / 卖出”，用户难以判断：

- 用了哪些数据？
    
- 哪些模型和工具被调用？
    
- 哪些证据支持？
    
- 哪些证据反对？
    
- 哪些规则或护栏被触发？
    
- 结论适合什么投资风格？
    
- 什么情况下结论失效？
    
- 哪一步最不可靠？
    

本产品必须让用户能查看完整 Decision Trace。

### 6.5 用户投资风格难以表达

不同用户的投资流程不同：

- 价值投资者关注安全边际；
    
- 成长投资者关注增速和预期上修；
    
- 趋势交易者关注价格行为；
    
- 事件驱动投资者关注催化剂；
    
- 逆向投资者关注情绪过度；
    
- 宏观配置型投资者关注利率、流动性和行业轮动。
    

产品不应强行输出一个统一结论，而应支持不同 Investment Playbook 下的决策分析。

### 6.6 难以复盘和改进

投资决策需要复盘：

- 当时为什么买？
    
- 当时为什么没买？
    
- 当时哪些证据被高估？
    
- 哪些风险被忽略？
    
- 哪些模型输出误导了判断？
    
- 哪个 Workflow 步骤需要调整？
    
- 哪个 Playbook 条件需要修订？
    
- 当前案例和历史案例有何相似？
    

因此，产品必须保存结构化的分析结果、执行日志和决策链路。

## 7. 核心理念

### 7.1 不以 Agent 为中心，而以投研任务和执行路径为中心

本产品不假设所有分析职能都必须由 Agent 完成。

更准确的原则是：

> 根据任务性质，选择最合适的执行机制。

可选机制包括：

- 确定性计算；
    
- 数据查询；
    
- 规则检查；
    
- 统计模型；
    
- 机器学习模型；
    
- 单次 LLM 调用；
    
- 固定 Workflow；
    
- Agent 推理；
    
- 多 Agent 辩论；
    
- Evaluator 审查；
    
- 人工确认节点。
    

Agent 是系统的一部分，但不是所有问题的默认解法。

### 7.2 模型和工具生成证据，Workflow 和 Agent 执行分析

系统应区分不同职责：

```text
数据连接器：获取上下文；
模型和工具：生成事实、指标、分类、评分、预测和异常检测；
Workflow：执行稳定、可重复的分析步骤；
Agent：处理开放式判断、解释、归因、冲突分析和反方观点；
Evaluator：审查输出质量；
Guardrails：控制风险和边界；
Decision Trace：记录证据链和执行链。
```

### 7.3 Skill 是能力封装之一，不是唯一核心

Skill 的合理定位是：

> 一种可复用的能力包形式，用于封装领域方法、工具使用说明、分析步骤、输出模板、案例和质量检查规则。

但产品的核心资产不只是 Skill，而是：

- 投资工作流；
    
- 分析能力包；
    
- 数据和模型接口；
    
- Investment Playbook；
    
- 案例记忆；
    
- 评估集；
    
- Guardrails；
    
- Decision Trace；
    
- 用户反馈和复盘记录。
    

### 7.4 不做黑箱多信号合成

本产品不默认把宏观、基本面、事件、情绪、技术面等信号粗暴加权成一个总分。

原因是：

- 不同信号时间周期不同；
    
- 不同信号置信度不同；
    
- 不同信号适用投资风格不同；
    
- 信号之间可能存在真实矛盾；
    
- 简单加权可能掩盖关键风险。
    

本产品更倾向于输出：

- 不同投资风格下的建议；
    
- 用户 Investment Playbook 下的通过 / 不通过结果；
    
- 决策链中的关键冲突；
    
- 支持证据和反方证据；
    
- 结论的适用条件和失效条件。
    

### 7.5 用户保留最终判断权

本产品不替代用户做最终投资决策。

系统输出的建议应定位为：

- 分析辅助；
    
- 决策检查；
    
- 反方提醒；
    
- 流程执行结果；
    
- 复盘记录。
    

用户始终保留最终判断权。

## 8. 产品总体架构

v0.2 将系统抽象为六层架构：

```text
1. Context Layer：数据与上下文层
2. Capability Layer：模型、工具与能力包层
3. Orchestration Layer：任务编排层
4. Execution Layer：Workflow 与 Agent 执行层
5. Governance Layer：护栏、评估、人工介入与可观测性层
6. Product Layer：用户流程、结果展示与 Decision Trace 层
```

---

## 9. Context Layer：数据与上下文层

Context Layer 负责提供系统分析所需的事实和上下文。

### 9.1 数据类型

包括但不限于：

- 行情数据；
    
- 财务数据；
    
- 公司公告；
    
- 公司新闻；
    
- 宏观数据；
    
- 行业数据；
    
- 政策文本；
    
- 地缘新闻；
    
- 社媒文本；
    
- 新闻标题；
    
- 分析师评级；
    
- 期权数据；
    
- 资金流数据；
    
- 用户持仓；
    
- 用户历史分析记录；
    
- 用户投资偏好；
    
- 历史案例库。
    

### 9.2 设计原则

Context Layer 的核心原则是：

1. 数据来源可追溯；
    
2. 数据时间戳清晰；
    
3. 数据质量可标记；
    
4. 原始事实与模型推断分离；
    
5. 缺失数据必须显式说明；
    
6. 不可靠数据不得被包装成确定事实。

## 10. Capability Layer：模型、工具与能力包层

Capability Layer 负责提供可调用的分析能力。

### 10.1 能力类型

包括：

- 财务指标模型；
    
- 估值模型；
    
- 技术指标模型；
    
- 情绪分类模型；
    
- 事件识别模型；
    
- 宏观 regime 模型；
    
- 风险模型；
    
- 新闻检索工具；
    
- 财报解析工具；
    
- 公告解析工具；
    
- 社媒分析工具；
    
- 数据校验工具；
    
- 代码执行工具；
    
- LLM 推理能力；
    
- Skill；
    
- Playbook；
    
- Case Library；
    
- Output Schema；
    
- Guardrail Rule；
    
- Evaluator。
    

### 10.2 Capability Package

Capability Package 是系统中可复用的能力单元。

一个 Capability Package 可以包含：

```text
1. 能力名称
2. 适用任务
3. 输入 schema
4. 输出 schema
5. 可调用工具
6. 可调用模型
7. 分析方法说明
8. Skill / prompt / instruction
9. 相关案例
10. 质量检查规则
11. 失败模式
12. Guardrails
13. Eval cases
```

### 10.3 Skill 的定位

Skill 是 Capability Package 的一种组成形式。

Skill 可以封装：

- 分析步骤；
    
- 工具使用说明；
    
- 领域知识；
    
- 判断规则；
    
- 输出模板；
    
- 案例；
    
- 禁止行为；
    
- 质量检查。
    

但 Skill 不直接等于产品核心，也不承担任务编排、状态管理、评估、护栏和用户体验的全部职责。

## 11. Orchestration Layer：任务编排层

Orchestration Layer 是 v0.2 的核心层。

它负责决定：

```text
哪些模块要运行；
运行顺序是什么；
哪些可以并行；
哪些需要路由；
哪些任务适合固定 workflow；
哪些任务适合 Agent 推理；
哪些任务需要多 Agent 辩论；
哪些任务需要 Evaluator 审查；
哪些任务需要人工确认；
哪些风险需要触发 Guardrail；
哪些输出需要重试或降级。
```

### 11.1 常见编排模式

系统应支持以下编排模式：

#### A. Fixed Workflow

适合稳定、可重复的任务。

例如：

```text
获取财务数据 → 计算指标 → 估值分位 → 生成基本面摘要
```

#### B. Prompt Chaining

适合需要分步骤处理的任务。

例如：

```text
公告提取 → 事件分类 → 影响路径分析 → 反方证据检查 → 输出结论
```

#### C. Routing

适合不同输入走不同模块。

例如：

```text
用户问财报 → Fundamentals Workflow
用户问突发新闻 → Event Workflow
用户问买点 → Technical Workflow
用户问长期配置 → Macro/Meso Workflow
```

#### D. Parallelization

适合多视角同时分析。

例如：

```text
Macro/Meso、Fundamentals、Event、Sentiment、Technical 五个能力域并行生成分析。
```

#### E. Orchestrator-Workers

适合复杂深度研究。

例如：

```text
中央编排器拆解任务，并动态分派给不同能力包、工具或 Agent。
```

#### F. Evaluator-Optimizer

适合需要质量审查的输出。

例如：

```text
分析 Agent 生成结论 → Evaluator 检查证据、逻辑、反方观点和不确定性 → 必要时要求重写。
```

#### G. Debate / Adversarial Review

适合存在明显分歧的投资判断。

例如：

```text
Bull Agent 提出正方观点；
Bear Agent 提出反方观点；
Risk Reviewer 检查关键风险；
Portfolio Reviewer 给出最终建议边界。
```

#### H. Human-in-the-loop

适合高风险或数据不充分场景。

例如：

```text
重大公司事件；
数据来源冲突；
模型置信度低；
建议涉及高风险交易；
用户设置要求人工确认。
```

### 11.2 编排原则

编排器应遵守以下原则：

1. 能用确定性流程解决的，不强行使用 Agent；
    
2. 能用模型或工具产生证据的，不让 Agent 编造；
    
3. 开放式判断交给 Agent；
    
4. 高风险结论必须经过审查；
    
5. 数据不足时必须降级或提示用户；
    
6. 关键执行路径必须进入 Decision Trace；
    
7. 用户可以看到系统为什么选择某条执行路径。
    

### 11.3 编排器与能力域的接口

> 本小节为 v0.3 新增。编排模式详细定义将迁移至 SPEC-007，五个能力域的完整输入输出将迁移至 SPEC-004。当前 §11.1 和 §12.2–§12.6 的内容将在后续版本中精简。

本产品采用分层编排设计。

编排器负责调度全局 Workflow 节点和分析能力域，但不直接侵入每个能力域的内部实现。

编排器可以直接调度的全局节点包括：

1. 任务解析；
2. 用户 Playbook 加载；
3. 上下文构建；
4. 冲突检测；
5. Playbook Evaluation；
6. Guardrail Check；
7. Decision Candidate 生成；
8. Decision Trace 构建。

对于五个分析能力域，编排器只调度能力域级别的 **Analysis Domain Job**。

五个分析能力域包括：

1. Macro / Meso；
2. Fundamentals；
3. Company Event / Catalyst；
4. Sentiment；
5. Technical / Market。

每个能力域内部如何组合模型、工具、固定 Workflow、Agent、Evaluator 和能力包，由该能力域自行定义。编排器只依赖能力域暴露的标准接口。

能力域标准接口将在 SPEC-004 中定义。

五个能力域各自返回 Analysis Card 后，由编排器的 **Playbook Evaluation** 节点结合用户 Investment Playbook，对五张 Analysis Card 进行综合判断，生成最终 **Decision Candidate**。这一综合机制不是简单加权，而是基于 Playbook 定义的条件、冲突规则和权重偏好进行结构化评估。具体机制由 SPEC-006（Investment Playbook）和 SPEC-007（Orchestration）共同定义。

## 12. Execution Layer：Workflow 与 Agent 执行层

Execution Layer 负责实际完成分析任务。

### 12.1 五个分析能力域

本产品保留五个核心分析能力域：

```text
1. Macro / Meso
2. Fundamentals
3. Company Event / Catalyst
4. Sentiment
5. Technical / Market
```

需要注意：

> 这五个是分析能力域，不一定是五个独立 Agent。

每个能力域可以由模型、工具、Workflow 和 Agent 组合完成。

### 12.1A 能力域标准输出：Analysis Card

> 本小节为 v0.3 新增。完整 schema 将在 SPEC-004 中定义。

每个能力域应向编排器返回标准化的 **Analysis Card**，至少包含：

1. 能力域名称；
2. 核心结论；
3. 支持证据；
4. 反方证据；
5. 置信度；
6. 时间周期；
7. 数据质量标记；
8. 关键风险；
9. 失效条件；
10. 可追溯证据引用。

Analysis Card 是编排器进行 Playbook Evaluation、冲突检测和 Decision Trace 构建的基础单元。能力域内部可以产生更丰富的中间产物，但向编排器暴露的接口应保持此最小结构。

### 12.2 Macro / Meso 能力域

负责分析宏观和中观环境。

主要问题：

- 当前宏观状态如何？
    
- 利率、通胀、就业、流动性、汇率、商品价格如何影响风险偏好？
    
- 行业政策、产业周期、监管变化对目标公司有何影响？
    
- 当前环境对估值扩张还是估值收缩更友好？
    
- 哪些行业受益，哪些行业承压？
    

典型执行方式：

```text
宏观数据模型
+ 政策/新闻检索工具
+ 行业映射模型
+ Macro/Meso Agent 解释
+ Evaluator 审查
```

典型输出：

- 宏观状态判断；
    
- 行业影响判断；
    
- 风险偏好判断；
    
- 对估值的影响；
    
- 对目标股票的正负面影响；
    
- 置信度；
    
- 失效条件。

- 估值是否合理？

```text

+ 估值模型
+ 公告/财报解析工具
+ Fundamentals Workflow
```

### 12.4 Company Event / Catalyst 能力域

负责分析公司相关事实性事件和催化剂。

主要问题：

- 公司最近发生了什么重要事件？

- 是否需要触发人工复核？
    

典型执行方式：

```text

+ 事件分类模型
+ 重要性评分
```

典型输出：

- 置信度；
    
- 失效条件。

## 13. Investment Playbook

Investment Playbook 是用户或系统定义的投资决策手册。

它不是简单 prompt，也不等同于 Skill。

它面向用户和产品层，表达：

```text
用户是谁；
投资风格是什么；
适合什么资产；
关注什么时间周期；
必须满足什么条件；
哪些情况禁止交易；
如何处理冲突信号；
何时需要人工确认；
最终建议如何表达。
```

### 13.1 Playbook 示例

#### 成长股中期持有 Playbook

```text
1. 公司收入增速必须高于行业中位数；
2. 毛利率不能连续两个季度恶化；
3. 自由现金流恶化时必须降级；
4. 估值高于历史 80% 分位时，不允许强买入；
5. 情绪极度过热时，只允许持有或等待；
6. 技术面跌破中期趋势时，必须触发风险提示；
7. 宏观环境压制估值时，应降低仓位建议；
8. 任何建议都必须列出失效条件。
```

#### 价值投资 Playbook

```text
1. 优先关注现金流质量和资产负债表稳健性；
2. 估值必须相对历史或同业有安全边际；
3. 不因短期情绪改善而忽视长期基本面；
4. 周期股必须明确所处周期位置；
5. 财务红旗出现时，必须降级；
6. 如果无法解释估值折价原因，不允许直接买入。
```

#### 趋势交易 Playbook

```text
1. 趋势必须明确；
2. 成交量需要确认突破或反转；
3. 不在波动率极端扩张时追高；
4. 必须给出失效价位；
5. 基本面仅作为过滤项，不作为主要入场依据；
6. 情绪过热时降低追踪优先级。
```

### 13.2 Playbook 与其他概念的区别

|概念|面向对象|作用|
|---|---|---|
|Skill|Agent / Capability Package|说明某项能力如何执行|
|Playbook|用户 / 产品 / 投资流程|定义用户如何做投资判断|
|Workflow|系统执行层|定义任务如何被执行和编排|
|Guardrail|风控和治理层|定义哪些行为或输出不允许|
|Decision Trace|用户信任层|记录系统如何得出结论|

### 13.3 Playbook 执行机制

> 本小节为 v0.3 新增。Playbook 的完整结构和执行机制将在 SPEC-006 中定义。

Investment Playbook 不等同于 prompt，也不等同于 Skill。

Playbook 的底层执行机制可以包括以下一种或多种方式：

1. 结构化规则（规则引擎逐条判定）；
2. Workflow 参数（控制编排器行为和路径选择）；
3. LLM 解释（由 Agent 基于 Playbook 文本做出判断）；
4. Evaluator 判断标准（作为输出审查的评判依据）；
5. Guardrail 条件（触发硬性边界）；
6. 用户偏好配置（风险偏好、仓位约束、持有周期）；
7. 历史案例参照（匹配相似情境的历史判断）。

SPEC-001 不决定 Playbook 的最终执行实现。

MVP 阶段优先验证用户是否认可"按某种投资手册生成透明投资判断"的产品价值，不强制支持用户自由编辑复杂 Playbook。

### 13.4 Playbook 适用性局限

> 本小节为 v0.3 新增。此局限为已知系统边界。

系统早期版本只能判断某个资产是否符合指定 Playbook，但不能充分判断**该 Playbook 本身是否适合当前市场环境**。

例如，一个成长股 Playbook 在低利率、高流动性环境下可能更适用；在加息周期、流动性收缩、估值压缩环境下可能过于严格，导致长期无可投资标的。系统可以报告"当前无标的通过 Playbook"，但无法主动提示"你的 Playbook 在当前 regime 下可能过于严格"。

未来系统需要支持 Playbook Applicability Evaluator，用于判断：

1. 当前市场 regime 是否适合该 Playbook；
2. 该 Playbook 是否过严或过松；
3. 用户是否因 Playbook 条件导致系统性错过机会；
4. Playbook 是否需要根据市场环境调整；
5. 当前结论是资产不符合 Playbook，还是 Playbook 本身需要复审。

该能力不进入 MVP，但应作为后续版本的重要方向，并在 SPEC-006 中预留扩展点。

## 14. Governance Layer：护栏、评估、人工介入与可观测性

Governance Layer 负责确保系统输出可信、可控、可审计。

### 14.1 Guardrails

系统必须具备以下护栏：

- 不承诺收益；
    
- 不输出无依据的强买入或强卖出；
    
- 数据不足时必须说明；
    
- 模型预测不得被表述为确定事实；
    
- 高风险建议必须标注风险；
    
- 涉及实盘交易时必须提醒用户自行决策；
    
- 不自动下单；
    
- 不绕过用户确认；
    
- 不在关键数据缺失时强行给明确结论；
    
- 不隐藏反方证据。
    

### 14.2 Evaluation

系统需要评估：

- 输出是否遵守结构；
    
- 是否引用了正确数据；
    
- 是否区分事实、模型输出和 Agent 判断；
    
- 是否列出反方证据；
    
- 是否表达不确定性；
    
- 是否符合 Playbook；
    
- 是否触发必要 Guardrails；
    
- 是否出现幻觉或无依据判断；
    
- 用户是否认为结果有帮助。
    

### 14.3 Human-in-the-loop

以下场景应触发人工确认或人工复核：

- 数据源冲突；
    
- 公司重大突发事件；
    
- 模型置信度低；
    
- 结论强烈但证据薄弱；
    
- 用户设置要求人工确认；
    
- 分析涉及高风险资产或高杠杆交易；
    
- 系统建议与用户 Playbook 明显冲突；
    
- 关键数据缺失但用户仍要求结论。
    

### 14.4 Observability

系统应记录：

- 每次任务的执行路径；
    
- 调用的模型；
    
- 调用的工具；
    
- 输入和输出；
    
- 中间状态；
    
- 错误和重试；
    
- Guardrail 触发记录；
    
- Evaluator 结果；
    
- 用户反馈；
    
- 成本、延迟和失败率。
    

### 14.5 Decision Trace 与 Observability 的关系

> 本小节为 v0.3 新增。

系统底层应维护统一 **Event Log**，用于记录任务执行过程中的关键事件。

Event Log 可以服务两个上层用途，二者的受众、展示目标、筛选规则和信息粒度不同：

**Observability** 面向开发者、运维人员和系统评估人员。它关注：

1. 模型调用；
2. 工具调用；
3. 执行路径；
4. 延迟；
5. 成本；
6. 错误；
7. 重试；
8. Guardrail 触发；
9. Evaluator 结果；
10. 系统稳定性。

**Decision Trace** 面向投资用户和投研复盘。它关注：

1. 使用了哪些数据；
2. 哪些证据支持结论；
3. 哪些证据反对结论；
4. 哪些 Playbook 条件被触发；
5. 哪些风险或不确定性存在；
6. 哪些模块之间存在冲突；
7. 为什么系统给出当前建议；
8. 用户下一步应该检查什么。

Observability 和 Decision Trace 共享底层 Event Log，但二者面向不同受众、使用不同筛选规则、呈现不同信息粒度。

Decision Trace 的完整设计将在 SPEC-008 中定义。Observability 的系统设计将在 SPEC-008 中作为同一份规范的运维视角章节同步定义。

### 14.6 用户私有数据

> 本小节为 v0.3 新增。

系统可能使用用户私有数据，包括：

1. 用户持仓；
2. 用户交易记录；
3. 用户历史分析记录；
4. 用户自定义 Playbook；
5. 用户反馈；
6. 用户复盘笔记。

所有用户私有数据必须明确标记为 `user_private`。

用户私有数据不得被默认用于公共案例库、公共评估集或其他用户的模型上下文。

用户私有数据的权限、存储、调用和删除机制将在 SPEC-012（数据治理）中定义。

## 15. Decision Trace

Decision Trace 是产品可信度的核心。

每一次分析结果都必须可展开查看完整决策链。

### 15.1 Decision Trace 必须回答的问题

每次建议必须能回答：

```text
1. 用户输入了什么？
2. 系统选择了哪条执行路径？
3. 使用了哪些数据？
4. 调用了哪些模型和工具？
5. 每个模型或工具输出了什么？
6. 哪些 Workflow / Agent 参与了分析？
7. 使用了哪个 Investment Playbook？
8. 哪些 Playbook 条件被触发？
9. 哪些 Guardrails 被触发？
10. 哪些证据支持当前结论？
11. 哪些证据反对当前结论？
12. 哪些信息缺失？
13. 哪些历史案例被引用？
14. 当前结论适合什么投资风格？
15. 什么条件会让结论失效？
16. 用户下一步应该检查什么？
```

### 15.2 Decision Trace 的目标

Decision Trace 的目标不是展示冗长推理，而是展示可审计证据链：

- 数据来源；
    
- 模型输出；
    
- 工具调用；
    
- 编排路径；
    
- 分析判断；
    
- 反方证据；
    
- 不确定性；
    
- 结论边界；
    
- 后续观察条件。
    

### 15.3 Decision Trace 的展示层级

建议分为三层展示：

#### Level 1：结论摘要

面向快速阅读。

包括：

- 当前建议；
    
- 适用投资风格；
    
- 核心理由；
    
- 最大风险；
    
- 失效条件。
    

#### Level 2：模块分析

面向投研审阅。

包括：

- Macro / Meso；
    
- Fundamentals；
    
- Event / Catalyst；
    
- Sentiment；
    
- Technical / Market；
    
- 风险与冲突点。
    

#### Level 3：执行链路

面向复盘和审计。

包括：

- 数据来源；
    
- 模型输出；
    
- 工具调用；
    
- Workflow 路径；
    
- Agent 输出；
    
- Evaluator 结果；
    
- Guardrail 记录；
    
- 用户反馈。

## 16. MVP 初步边界

SPEC-001 不最终定义 MVP，但给出方向性边界。

### 16.1 MVP 用户定位

> 本小节为 v0.3 新增。

MVP 阶段不面向所有普通投资者，而优先面向具备主动研究习惯、关注宏观/中观环境与公司基本面关系的投资者。

典型 MVP 用户包括：

1. 资本周期投资者；
2. 产业周期投资者；
3. 中长期基本面投资者；
4. 价值与成长结合型投资者；
5. 关注行业供需、资本开支、政策环境和估值周期的专业个人投资者。

因此，Macro / Meso 在 MVP 中保留为一级能力域，但范围应受到控制。

MVP 阶段的 Macro / Meso 不做完整宏观研究系统，而聚焦：

1. Market Regime；
2. Industry Cycle；
3. Policy Context；
4. Capex Cycle；
5. Rate / Liquidity Context；
6. 对目标公司或行业估值环境的影响。

如果后续产品转向普通投资者市场，Macro / Meso 可以降级为背景能力。

### 16.2 MVP 应该验证什么

MVP 不应优先验证"用户是否愿意编辑 Skill"。

MVP 应优先验证：

> 用户是否认可一个可追踪、可解释、可复盘的 Agentic 投研工作流，比普通股票分析 Chatbot 更有价值。

### 16.3 MVP 应该包含

建议 MVP 聚焦：

```text
单股票 + 单投资风格 + 五个分析能力域 + 固定透明工作流 + Decision Trace
```

MVP 最小闭环：

1. 用户输入一个股票 ticker；
    
2. 用户选择一个 Investment Playbook；
    
3. 系统执行固定但透明的投研 Workflow；
    
4. 系统调用五个分析能力域；
    
5. 每个能力域输出结构化分析；
    
6. 系统生成投资判断建议；
    
7. 系统展示 Decision Trace；
    
8. 用户可以反馈：有帮助 / 没帮助 / 哪一步有问题；
    
9. 用户可以保存本次分析记录用于复盘。
    

### 16.4 MVP Evaluator 边界

> 本小节为 v0.3 新增。Evaluator 完整机制将在 SPEC-009 中定义。

MVP 阶段的 Evaluator 不做自动优化和自动重写。

Evaluator 只做检查、标注和提示。它可以标记：

1. 无证据判断；
2. 幻觉嫌疑；
3. 反方证据缺失；
4. 数据质量不足；
5. Playbook 条件遗漏；
6. 模块冲突未处理；
7. 结论过度自信；
8. 风险提示不足。

Evaluator 的输出进入 Decision Trace，供用户查看。

用户可以基于 Evaluator 标注主动要求系统修正，但系统不在 MVP 阶段默认自动重写分析结果。

### 16.5 MVP 可以暂不包含

MVP 阶段不做：

1. 自动下单；
    
2. 股票池大规模扫描；
    
3. 组合优化；
    
4. 实盘交易；
    
5. 复杂回测系统；
    
6. 多账户管理；
    
7. 面向机构的完整合规审批工作流；
    
8. 完整研报生成平台；
    
9. 社交社区功能；
    
10. 收益预测承诺；
    
11. 用户自由编辑复杂 Skill；
    
12. 用户自由搭建复杂 Agent Workflow。

## 17. 产品成功标准

### 17.1 用户价值标准

产品是否成功，首先看用户是否认为它帮助自己：

- 更快完成投资分析；
    
- 更清楚看见反方观点；
    
- 更稳定执行某种投资风格；
    
- 更少被单条新闻或情绪带偏；
    
- 更容易复盘过去的判断；
    
- 更清楚知道自己为什么买、为什么不买、为什么等待；
    
- 更信任系统的分析链路，而不是只看最终建议。
    

### 17.2 系统质量标准

系统质量不以“预测对了几次”为唯一标准，而应包括：

- 输出是否结构化；
    
- 证据是否可追溯；
    
- 执行路径是否清晰；
    
- 是否区分事实、模型输出和 Agent 判断；
    
- 是否展示反方证据；
    
- 是否表达不确定性；
    
- 是否给出失效条件；
    
- 是否符合 Investment Playbook；
    
- 是否触发必要 Guardrails；
    
- 是否支持用户复盘；
    
- 是否避免无依据买卖建议。
    

### 17.3 MVP 验证指标

MVP 可观察指标包括：

- 用户是否愿意重复分析同一股票；
    
- 用户是否愿意选择不同 Investment Playbook 对比结果；
    
- 用户是否频繁展开 Decision Trace；
    
- 用户是否认为反方证据有帮助；
    
- 用户是否把分析结果用于真实投研记录；
    
- 用户是否反馈“比普通 Chatbot 更可信”；
    
- 用户是否能指出某一步分析有问题，并希望系统修正；
    
- 用户是否愿意保存分析记录用于复盘。

## 18. 关键风险

### 18.1 数据风险

- 数据源不完整；
    
- 数据延迟；
    
- 数据错误；
    
- 新闻误报；
    
- 社媒噪声；
    
- 财务数据口径不一致；
    
- 模型基于错误数据生成结论。
    

### 18.2 模型风险

- 模型预测失效；
    
- 情绪分类错误；
    
- 事件分类错误；
    
- 宏观 regime 判断滞后；
    
- 统计模型过拟合；
    
- LLM 错误解释模型输出；
    
- 模型对近期市场 regime 不适应。
    

### 18.3 Agent 风险

- Agent 迎合模型结论；
    
- Agent 忽略反方证据；
    
- Agent 过度自信；
    
- Agent 把相关性解释为因果；
    
- Agent 在数据不足时强行给结论；
    
- Agent 输出听起来合理但不可验证的分析。
    

### 18.4 Orchestration 风险

- 编排路径错误；
    
- 不该使用 Agent 的地方使用了 Agent；
    
- 该触发审查时没有触发；
    
- 复杂任务被过度简化；
    
- 简单任务被过度复杂化；
    
- 多模块输出冲突但未被显式处理。
    

### 18.5 Playbook 风险

- Playbook 设计过于僵化；
    
- Playbook 不适配不同市场环境；
    
- 用户误配置投资流程；
    
- Playbook 和用户真实行为不一致；
    
- 系统过度迎合用户偏好，放大用户原有偏见。
    

### 18.6 产品风险

- 产品被用户误解为荐股工具；
    
- 用户只看最终建议，不看 Decision Trace；
    
- 过度复杂导致新用户难以上手；
    
- 数据和模型成本过高；
    
- 输出太长，降低使用效率；
    
- 系统解释能力强，但真实投资帮助有限。

## 19. 设计原则

### 19.1 透明优先

任何投资建议都必须能展开查看证据链和执行链。

### 19.2 编排优先

系统应根据任务性质选择最合适的执行路径，而不是默认所有任务都交给 Agent。

### 19.3 证据优先

模型、工具和数据应尽可能先生成可验证证据，Agent 不应凭空补全事实。

### 19.4 用户流程优先

系统不强加统一投资方法，而应尊重用户选择的 Investment Playbook。

### 19.5 反方证据必选

任何正面结论都必须包含反方证据，任何负面结论也必须包含可能的正面反转条件。

### 19.6 不确定性显式表达

数据不足、模型置信度低、结论依赖假设时，系统必须明确说明。

### 19.7 模型输出不等于事实

系统必须区分：

- 原始事实；
    
- 模型计算结果；
    
- 模型预测；
    
- Agent 判断；
    
- Playbook 结论；
    
- 用户最终决策。
    

### 19.8 不做强制统一评分

除非用户明确选择某种评分流程，否则系统不应默认把多模块结果合成为一个黑箱总分。

### 19.9 可复盘

每次分析都应保存可复盘记录，包括输入、模型输出、工具调用、执行路径、Agent 结论、Guardrail 触发和最终建议。

### 19.10 人类保留最终控制权

系统可以给出分析和建议，但不应替代用户做最终投资决策。

## 21. SPEC-001 v0.3 总结

本产品的核心不是“让 Agent 模拟投研团队”，也不是“用 Skill 指挥 Agent”。

本产品的核心是构建一个：

> 可编排、可审计、可评估、可复盘的 Agentic 投研决策工作流系统。

它通过数据、模型、工具、Workflow、Agent、Investment Playbook、Guardrails、Evaluation 和 Decision Trace 的组合，帮助用户更稳定、更透明地形成投资判断。

产品最重要的资产包括：

1. 高质量数据与上下文；
    
2. 可复用 Capability Package；
    
3. 可靠的 Orchestration Layer；
    
4. 五类分析能力域；
    
5. Investment Playbook；
    
6. Decision Trace；
    
7. Guardrails 与 Evaluation；
    
8. 用户反馈和复盘记录。
    

本 SPEC 作为产品定义与边界文档，后续所有功能设计、模块设计、MVP 设计都应围绕这一核心定义展开。
